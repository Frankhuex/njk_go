# 离线记忆生产任务设计

## 1. 目标

本文档描述一个新的离线任务入口，用于从历史 `message` 表中按批读取旧消息，复用当前项目已经实现的“标准记忆提取、embedding、去重、写库”流程，持续生产旧记忆到：

- `memory_fact`
- `memory_impression`

本设计只写方案，不落代码。

## 2. 设计目标

需要满足以下要求：

- 提供一个新的可执行入口 `main.go`
- 不要求传入日期范围参数
- 首次运行时默认从每个群组最早消息开始，到任务启动时刻为止
- 任务启动时要记录“本次运行的结束时间戳”，避免运行过程中新消息不断加入导致扫描范围漂移
- 使用 JSON 状态文件记录进度，便于重启恢复和人工复盘
- 每个群组一个 goroutine 并行生产
- 每分钟调度一次
- 每次每个群最多取 100 条消息
- 对取出的消息走“标准记忆召回、分析、存储流程”
- 将结果写入数据库

## 3. 推荐入口与脚本

### 3.1 新入口

建议新增：

- `cmd/memory-factory/main.go`

原因：

- 当前在线服务入口已经迁到 `cmd/server/main.go`
- `gen_gorm` 也已迁到 `cmd/gen-gorm/main.go`
- 新的离线记忆生产任务继续放在 `cmd/` 下最统一

### 3.2 新脚本

建议不再新增独立脚本，而是统一复用：

- `run.sh --memory`
- `build.sh --memory`

## 4. 运行方式

### 4.1 启动方式

建议直接支持：

```bash
sh run.sh --memory
```

语义：

- 如果没有已有状态文件：
  - 从每个群最早消息开始
  - 截止到“任务启动时刻”
- 如果已有状态文件：
  - 从 JSON 状态文件中恢复各群当前进度
  - 继续处理到本次任务启动时刻

### 4.2 JSON 状态文件

推荐使用：

- `runtime/memory-backfill/state.json`

并配合：

- `runtime/memory-backfill/history/`

用于历史运行归档。

## 5. 为什么要记录任务启动时刻

如果任务运行时不先固定一个“本次扫描上界”，会有两个问题：

- 任务跑的过程中，新的在线消息不断写入 `message`
- 扫描窗口会不断向后延伸，导致一次回填任务没有稳定边界

因此建议：

- 任务一启动，立即记录 `runEndAt := time.Now()`
- 本次任务始终只处理 `message.time <= runEndAt` 的消息

这样：

- 一次运行的边界固定
- 结果可复现
- 更适合断点续跑

## 6. JSON 状态文件设计

推荐使用一个结构化 JSON 文件来替代数据库进度表。

原因：

- 比纯 `txt` 更结构化
- 机器读取和恢复更稳定
- 人工查看和排障仍然足够方便
- 当前阶段比新增数据库进度表更轻量

推荐结构示意：

```json
{
  "task_name": "memory_backfill",
  "started_at": "2026-07-08T18:10:00+08:00",
  "run_end_at": "2026-07-08T18:10:00+08:00",
  "status": "running",
  "groups": {
    "1050660050": {
      "status": "running",
      "last_message_time": "2026-06-01T12:30:15+08:00",
      "last_message_id": "988460475",
      "processed_messages": 300
    },
    "123456789": {
      "status": "done",
      "last_message_time": "2026-06-03T09:10:11+08:00",
      "last_message_id": "988460520",
      "processed_messages": 1000
    }
  }
}
```

建议记录字段：

- 任务级：
  - `task_name`
  - `started_at`
  - `run_end_at`
  - `status`
- 群级：
  - `status`
  - `last_message_time`
  - `last_message_id`
  - `processed_messages`

推荐规则：

- 启动时：
  - 若 `state.json` 不存在，初始化新文件
  - 若存在，则读取最新游标并恢复
- 运行中：
  - 每次某群成功推进一批后，更新对应群的游标
- 全部完成后：
  - 保留 `state.json` 作为最新游标
  - 同时将当前状态快照归档到 `history/`

## 7. 与现有记忆流程的关系

当前项目里，在线记忆流程已经具备以下能力：

- 消息窗口聚合
- 记忆提取
- embedding
- 去重
- 落库

因此离线任务不应重新发明一套逻辑，而应尽量复用现有 `service` 中的标准记忆生产链。

推荐原则：

- 在线和离线共用一套“窗口 -> 提取 -> embedding -> 去重 -> 写库”逻辑
- 离线任务只负责“读取历史消息并喂入这套流程”
- 不要复制一套新的记忆提取代码

## 8. 推荐架构

### 7.1 入口层

`cmd/memory-factory/main.go` 负责：

1. `config.Load()`
2. `pgstore.InitStore(cfg.DSN())`
3. 初始化 `ai.NewClient(...)`
4. 初始化 `service.NewService(...)`
5. 读取或初始化 `state.json`
6. 记录 `runEndAt`
7. 启动离线记忆生产任务

### 7.2 service 层

建议在 `internal/service/` 新增离线任务相关文件，例如：

- `memory_backfill.go`
- `memory_backfill_runner.go`

主要职责：

- 组织群级并发
- 驱动每分钟调度
- 把历史消息转成待处理窗口
- 调用已有的标准记忆流程

### 7.3 pgstore 层

需要新增几类数据库方法：

- 枚举有历史消息的群
- 查询某群最早消息时间
- 按群分页读取历史消息
- 不需要负责进度表读写，进度由 JSON 状态文件承担

## 9. 为什么还要存 `last_message_id`

只存时间不够稳，因为：

- 同一秒内可能有多条消息
- 只按时间推进可能导致重复或遗漏

更稳的方式是双游标：

- `last_message_time`
- `last_message_id`

分页条件可写成：

- `time > last_time`
- 或 `time = last_time AND message_id > last_message_id`

## 10. 分群并发模型

### 9.1 基本模型

每个群组一个 goroutine。

流程：

1. 先查出有历史消息的群 ID 列表
2. 为每个群启动一个 goroutine
3. 每个 goroutine 独立维护该群的回填进度
4. 每分钟触发一次该群的拉取和生产

### 9.2 为什么按群并发是合理的

- 当前记忆窗口已经按群聚合
- 不同群之间完全可以并行
- 群内顺序必须稳定，因此群内不建议再开多 worker 抢同一段历史

### 9.3 并发控制建议

虽然逻辑上是“每群一个 goroutine”，但仍建议保留全局并发限流能力。

例如：

- 如果群太多，仍可以用 semaphore 控制同时真正执行 LLM/embedding 的群数量

设计上可以分为两层：

- 逻辑层：每群一个 goroutine
- 资源层：统一的 AI 并发限流

## 11. 每分钟调度一次的含义

用户要求“每分钟运行一次，取出 100 条该群消息”。

建议解释为：

- 每个群 goroutine 内部有一个 `ticker := time.NewTicker(1 * time.Minute)`
- 每到一分钟：
  - 读取该群尚未处理的下一批最多 100 条消息
  - 组织成记忆窗口
  - 走标准记忆流程
  - 更新进度

### 10.1 启动后是否立即跑一次

建议：

- 启动后先立即执行一次
- 然后再进入每分钟 ticker

否则任务启动后还要空等一分钟。

## 12. 每批 100 条如何组织成窗口

### 11.1 读取粒度

数据库层每次取：

- 某群
- 某进度之后
- 最多 100 条
- 且 `time <= runEndAt`

### 11.2 窗口组织方式

当前在线记忆逻辑是：

- 待记忆队列按群聚合
- 满 5 条或超时后触发

离线任务建议沿用相同思路，但不必真的依赖在线 ticker。

推荐做法：

- 读出 100 条历史消息
- 顺序喂给一套“离线窗口构造器”
- 每累计 5 条形成一个窗口
- 剩余不足 5 条时：
  - 如果本批已到该群范围末尾，也要强制 flush
  - 如果后面还有消息，可带到下一批继续攒

### 11.3 窗口内容

窗口里必须保留多人上下文：

- `group_id`
- `message_id`
- `user_id`
- `actor_name`
- `content`
- `time`
- `is_bot_reply`

聚合文本建议继续用类似：

- `昵称(user_id): 内容`

让 LLM 自己判断：

- 这条记忆是否只属于一个用户
- 是否应写 `user_id`
- 是否应写成群公共事实

## 13. 标准记忆流程如何复用

离线任务建议复用现有标准流程的以下阶段：

1. 窗口文本过滤
2. LLM 提取候选记忆
3. embedding
4. hash 去重
5. 相似度去重
6. 写入 `memory_fact` / `memory_impression`

### 12.1 建议抽象

推荐把当前在线记忆生产进一步抽成一个明确的 service 能力，例如概念上收敛为：

- `ProcessMemoryWindow(...)`

在线：

- 消息队列触发后调用它

离线：

- 历史回填窗口触发后也调用它

这样最不容易出现在线/离线行为漂移。

## 14. 首次运行与恢复规则

### 14.1 首次运行

如果没有 `state.json`：

- 每个群从该群最早消息开始
- 截止到本次任务启动时刻 `runEndAt`

这要求 `pgstore` 提供：

- 查询某群最早消息时间

### 14.2 恢复运行

如果存在 `state.json`：

- 按群恢复 `last_message_time + last_message_id`
- 本次任务的上界仍然固定为本次新启动时刻 `runEndAt`
- 对已经没有新增消息的群，直接跳过，不再启动 goroutine

### 14.3 完成后归档

当前轮全部完成后：

- 把 `state.json` 归档到 `history/时间戳.json`
- 但仍保留 `state.json` 作为下一次增量启动的最新断点

## 15. 新增 pgstore 能力建议

建议新增以下方法。

### 15.1 群枚举

- 查询有哪些 `group_id` 拥有历史消息

### 15.2 最早消息

- 查询某群最早消息时间

### 15.3 历史分页读取

- 查询某群在 `runEndAt` 边界内、指定游标之后的最多 100 条消息

返回内容建议尽量复用现有 `StoredMessage` 结构，或新增专门的历史窗口结构。

## 18. 失败处理

### 17.1 单群失败不应拖垮全局

某一个群失败时：

- 只记录该群失败
- 不影响其他群继续生产

### 17.2 单批失败如何处理

建议：

- 当前批失败时，不推进游标
- 下一轮重试同一批

这样最保守，避免漏数据。

### 17.3 幂等性

因为记忆写入本身已有：

- hash 去重
- 相似度去重

所以即便重试同一批，最终结果也可接受，但仍建议依赖 JSON 状态文件尽量减少重复成本。

## 19. 为什么不建议单次全量扫到底

虽然也可以写成一个“从头到尾跑完退出”的任务，但当前要求更像一个节流生产器：

- 每分钟
- 每群
- 最多 100 条

这种模式更有几个好处：

- 不会瞬间把 LLM/embedding 打满
- 可以与在线服务共存
- 适合大群旧数据慢慢回填

## 21. 与当前在线记忆队列的关系

在线队列解决的是：

- 新消息实时进入
- 群内按窗口聚合

离线回填任务解决的是：

- 历史消息补生产

两者不应直接共用同一个 runtime 队列对象，但应共用同一套窗口处理逻辑。

建议关系如下：

- 在线：`pendingMemoryQueue` 负责触发
- 离线：历史批读取器负责触发
- 两者最终都调用同一个“窗口记忆生产”方法

## 23. 推荐结论

这个需求完全能做，而且和当前项目结构是兼容的。

推荐落地方式是：

- 新建 `cmd/memory-factory/main.go`
- 通过 `run.sh --memory` / `build.sh --memory` 启动与构建
- 使用 `runtime/memory-backfill/state.json` 持久化状态
- 按群并发
- 每分钟每群取 100 条
- 启动时固定 `runEndAt`
- 首次运行默认从每群最早消息开始
- 后续运行默认从上次最新游标继续
- 通过 JSON 状态文件持久化断点
- 复用现有标准记忆提取、embedding、去重、写库流程

核心原则只有一条：

- 不要再写一套独立的“离线记忆逻辑”
- 只新增“历史消息读取 + 调度 + 断点恢复”
- 记忆分析和存储必须与在线流程共用同一标准链路

这样后续无论是在线实时生产还是离线旧消息补生产，最终的记忆行为都会保持一致。
